## Step 10.2 — Concrete hybrid-AI plan (BT shell + Utility scoring)

Supersedes the "step layout" sketch at the bottom of
[10-long-term-ai.md](10-long-term-ai.md). Same shape (utility inside a
BT shell) but with real class names, file paths, data schemas, and the
specific reference implementations we're porting from.

## Source material we're drawing on

Two read-only reference codebases live at `reference-code/`
(git-ignored; extracted from unrelated projects). Do not depend on
them directly — they're scaffolding and the ideas migrate into
Ccgnf namespaces. Specifically:

- `reference-code/BehaviorTree/` — a stateless BT runner. Key ideas
  we'll reuse: `BtNode` record (Composite/Decorator/Leaf union),
  `BtStatus { Success, Failure }` (no Running — every tick re-enters
  the tree), `IBtContext` bridging variables + actions, JSON
  serialisation via `BtSerializer`, the `Spec` pattern (type-safe
  condition builders producing the same strings the runtime parses),
  and `TreeLog` (JSONL append log with generation + parent metadata).
- `reference-code/utilityAI/` — per-action scoring. Key ideas we'll
  reuse: `Score(profile, state, action, world)` returning a float,
  normalised property values fed through response curves
  (piecewise-cubic Hermite with a linear fast path when tangents are
  zero), requirement gates evaluated before scoring, deterministic
  noise keyed on `(autonomeId, actionId, tick)`, and a
  "vital zero-lock" fallback (under collapse conditions, restrict the
  candidate pool to actions that actually address the critical
  property, with a graceful re-run if that pool is empty).

Neither fits Resonance as-is — both assume a persistent agent in a
continuous world. The port strips state: a CCGNF `InputRequest`
exposes a fixed list of `LegalAction`s, scoring happens once, the bot
returns one `RtSymbol`, done. No cooldowns. No tick counter across
decisions. Seeded randomness only where ties need breaking.

## Target shape

```
┌─────────────────────────────────────────┐
│   Room.ChooseCpuAction (existing)       │
│   ↓                                     │
│   IRoomBot.Choose(run, pending, id)     │
│   ↓                                     │
│   UtilityBot:                           │
│     1. PhaseBtRunner selects Intent     │
│        from top-level BT (4 states)     │
│     2. Intent loads weight table        │
│     3. UtilityScorer scores every       │
│        LegalAction                      │
│     4. Highest score wins; ties broken  │
│        by seeded hash                   │
└─────────────────────────────────────────┘
```

The BT is **intent selection only**. Leaves produce an intent enum —
they never "perform an action" the way the reference BT's
`ExecuteAction` does. Every real move goes through the utility scorer
so the weighting is transparent.

## Phase breakdown

### 10.2a — Extract `IRoomBot` (no behaviour change)

Lift `Room.ChooseCpuAction` + its two helpers (`PickTargetForCpu`,
`PickArenaForCpu`) into `FixedLadderBot`. `Room` keeps an
`IRoomBot _bot` field; default construction wires the fixed ladder so
tests + the human-vs-CPU room flow are unaffected.

```csharp
public interface IRoomBot
{
    RtValue Choose(InterpreterRun run, InputRequest pending, int cpuEntityId);
}
```

Room constructor gets a `BotKind` option (`"fixed" | "utility"`); the
REST create-room endpoint accepts it, default `"fixed"` until 10.2e
ships. Commit at the end of this phase: parity tests green, the
existing 61afdb3-style playthrough still ends the same way byte-for-
byte.

### 10.2b — Utility skeleton that wraps the ladder

`UtilityBot` implements `IRoomBot` but initially delegates to
`FixedLadderBot` through a single `LadderOverrideConsideration`
with weight 100. Every other consideration is registered but weighted
0. The point: prove the plumbing works end-to-end (ScoringContext
builds, top-3 logging fires, SSE frames emit) before any new
judgement is introduced.

`ScoringContext` is the single snapshot the scorer reads:

```csharp
public sealed class ScoringContext
{
    public GameState State { get; }            // read-only snapshot
    public InputRequest Pending { get; }
    public int CpuEntityId { get; }
    public Intent Intent { get; }
    public int Seed { get; }                   // room.seed + pending.Prompt.GetHashCode()
    public PlayerView Cpu { get; }             // cached lookups for the CPU seat
    public PlayerView Opponent { get; }
}
```

`PlayerView` caches hand count, aether, banner, conduit integrities —
things considerations would otherwise recompute repeatedly. Built once
per `Choose` call.

### 10.2c — Play-card considerations + weight file format

Ship the four play-card considerations promised in
[10-long-term-ai.md §10b](10-long-term-ai.md):

- `OnCurveConsideration` — Hermite curve over `|cost − aether|`.
  Knots: (0, 1.0), (1, 0.5), (2, 0.0). Linear segments, zero tangents
  so we hit the fast path in `CurveEvaluator`.
- `TempoPerAetherConsideration` — `force / max(cost, 1)` normalised
  against an upper bound of 4. Curve knots (0, 0), (1, 0.4),
  (2, 0.75), (4, 1.0).
- `ResonanceAlignmentConsideration` — reads
  `player.derived.banner` when present; falls back to iterating
  `zones["resonance_field"]` and counting faction overlap.
- `ClearHandConsideration` — +1.0 when hand is at `hand_cap - 1`,
  linearly down to 0 when hand has ≤ 3 cards. Lets the bot dump
  low-value cards rather than burn.

Weights file (`encoding/ai/utility-weights.json`):

```json
{
  "version": 1,
  "seed_tiebreak": true,
  "intents": {
    "early_tempo":    { "on_curve": 1.4, "tempo_per_aether": 1.0, ... },
    "pushing":        { "on_curve": 0.8, "resonance_alignment": 2.2, ... },
    "defend_conduit": { "on_curve": 0.6, "lowest_live_hp": 2.5, ... },
    "lethal_check":   { "conduit_softness": 3.0, ... },
    "default":        { "on_curve": 1.0, "tempo_per_aether": 1.0, ... }
  }
}
```

If the file is missing, `Weights/default.json` (embedded resource)
takes its place. A missing consideration key defaults to 0, not 1 —
an unspecified consideration is off, not average.

### 10.2d — Target + arena considerations

Port the remaining four from §10c, grounded in concrete counter
names we already read in `Room.PickTargetForCpu`:

- `LowestLiveHpConsideration` — inverse of min live counter over
  `{integrity, current_ramparts, current_hp}`. Below zero = ignore
  (never target collapsed entities).
- `OpponentPriorityConsideration` — hard category boost: Conduit =
  1.0, Unit = 0.6, Maneuver-token = 0.3, never self = 0.0.
- `OverlapConsideration` — used for `target_arena`. +1.0 where the
  opponent already has a Unit (forces Clash); +0.3 where we have a
  Unit with no opposing Unit (safe push).
- `ConduitSoftnessConsideration` — `1 - integrity/max_integrity`
  for the targeted arena's opponent conduit. Closes the game.
- `ThreatAvoidanceConsideration` — shallow, reactive: if declaring
  an attacker would leave our conduit in the same arena at integrity
  ≤ `opponent_unit_force`, score the attack at 0. This is the single
  piece of one-ply-look-ahead we allow. No cloning state; read force
  from cached `opponent_unit_force_by_arena`.

### 10.2e — Top-level BT shell

Four intent-selection states expressed as a `PhaseBtRunner` Selector.
The BT is loaded from `encoding/ai/phase-bt.json` so a designer can
edit the selection precedence without touching C#:

```json
[
  {
    "type": "Selector",
    "children": [
      {
        "type": "ConditionGate",
        "value": "opponent_standing_conduits == 1",
        "children": [{ "type": "Action", "value": "intent:lethal_check" }]
      },
      {
        "type": "ConditionGate",
        "value": "min_own_conduit_integrity <= 3",
        "children": [{ "type": "Action", "value": "intent:defend_conduit" }]
      },
      {
        "type": "ConditionGate",
        "value": "banner_matches_in_hand >= 2",
        "children": [{ "type": "Action", "value": "intent:pushing" }]
      },
      {
        "type": "ConditionGate",
        "value": "turn_number <= 3",
        "children": [{ "type": "Action", "value": "intent:early_tempo" }]
      },
      { "type": "Action", "value": "intent:default" }
    ]
  }
]
```

The `BtContext` implementation exposes the variables above via
`ResolveVariable` — exactly the pattern from
`reference-code/BehaviorTree/BtRunner.cs` lines 147–216. Actions set
a side-channel `ChosenIntent` on the context (instead of the
reference's `ExecuteAction(string)`) and return `Success`.

Priority of intents: first matching gate wins. No `Running` status —
every decision re-enters the tree from scratch. This is a conscious
simplification; if phase-level memory is needed later, a
`PhaseMemory` key on the room survives across `Choose` calls without
turning the BT itself stateful.

### 10.2f — Debug overlay + SSE frame

`CpuDecisionFrame` — serialised as a `RoomEventFrame` with
`EventType = "CpuDecision"` when `CCGNF_AI_DEBUG=1`:

```json
{
  "eventType": "CpuDecision",
  "field.intent": "pushing",
  "field.chosenLabel": "play_card:ember_pyre",
  "field.chosenScore": "3.42",

```

Web `play` tabletop already renders `RoomEventFrame`s in the right

Separately, every decision appends to
`logs/cpu-decisions-{roomId}.jsonl` (git-ignored) via a port of

`CpuDecisionFrame` per line plus the full consideration breakdown —
same shape the Autonome reference used for its tree history. This is
where playtest complaints turn into fixture cases.

### 10.2g — Sticky intent (across-decision memory)

The BT per-decision reset is fine for novel situations but produces
visible flip-flopping inside a single turn: the bot plays a tempo
card under `early_tempo`, the state updates, the next LegalAction
request re-enters the tree under `pushing`, and now the bot passes
to save aether for a Resonance push. The author sees "you just
played a tempo card, why are you now passing?"

Fix with a lightweight `PhaseMemory` on the `Room`:

```csharp
public sealed class PhaseMemory
{
    public Intent CurrentIntent { get; set; }
    public int IntentAgeSteps { get; set; }       // decisions since set
    public string ActivePhase { get; set; }       // Rise / Channel / Clash / Fall
}
```

BT's Selector evaluation gets a stickiness bias: if `CurrentIntent`
was chosen ≤ N decisions ago *and* `ActivePhase` hasn't changed,
multiply that intent's matching gate score by 1.5 before picking
the winner. The bias decays by 0.1 per decision; resets entirely
on phase transition. N defaults to 5, tunable in
`phase-bt.json` under a top-level `"stickiness"` block.

Crucially, the memory lives in `Room`, not in the bot itself. The
bot stays pure: `Choose(run, pending, cpuId, memory)`. This keeps
the scorer deterministic given its inputs, and makes memory easy
to inspect in the `CpuDecision` SSE frame for verification.

Lethal-check short-circuits stickiness: if `opponent_standing_
conduits == 1`, `lethal_check` always wins regardless of stickiness
bias. Defensive intent also short-circuits — a sudden threat to a
friendly conduit should always re-plan.

### 10.2h — Deck archetype tagging + "Suggest an AI" tournament

Deck creation (route `#/decks`) currently has `factions` but no
archetype tag. Add an **Archetype** field to the deck editor —
a multi-select chip input with a curated vocabulary
(`aggro`, `midrange`, `control`, `tempo`, `combo`, `disruption`,
`ramp`). Values write to `archetypes: [...]` in the deck JSON.
Existing decks default to `[]` and get a "no archetype set" warning
chip in the lobby.

Once decks carry archetype tags, the matchmaking problem from
§"Decks × AI" is automatable. Add a **"Suggest an AI"** button to
the deck detail view that runs a client-initiated tournament:

1. Frontend calls `POST /api/ai/tournament` with the deck id.
2. Server runs a mirror round-robin: the deck played by every
   registered bot profile against itself, N games each (default 20).
   Seeded for reproducibility.
3. Server optionally accepts `extraPairs: [{deckId, botProfile}, ...]`
   so the author can pin a few known-good combinations into the same
   tournament run for baseline comparison — e.g. include
   `(ember-aggro, v1-ember-aggro)` so the author sees how their new
   deck compares to an already-tuned pair.
4. Response streams progress as SSE: a `TournamentProgress` frame
   per completed match with `{ deckId, botProfile, winRate,
   avgTurns }`.
5. UI renders a sortable leaderboard. Top result = suggested AI;
   the button becomes "Use <profile>" and on click writes the
   affinity declaration to the deck JSON (`suggested_ai:
   "stable/v1-ember-aggro"`) and to the bot profile's
   `pairs_well_with` list.

Key design decisions:

- **Mirror first, then optional cross-deck.** Mirror matches
  isolate "how well does this AI pilot this deck"; cross-deck
  matches answer "how does this AI × this deck fare against real
  opposition." The tournament runs mirror by default because it
  answers the primary question (affinity suggestion) without a
  combinatorial explosion, and accepts `extraPairs` as an explicit
  expansion knob.
- **Streaming, not blocking.** A 4-bot × 20-game tournament is
  ~80 full playthroughs. Stream progress so the UI stays responsive
  and the author can cancel early if a clear leader emerges.
- **Tournament results are a benchmark artifact.** The response is
  also written to `logs/tournament-<deckId>-<timestamp>.json` for
  later comparison. If a bot is retuned, re-running the same
  tournament produces a diff the author can read.
- **Reuses `make bench-ai`.** The tournament endpoint is a thin
  wrapper over the existing benchmark harness (§10.2i) with the
  deck and bot list as parameters. No parallel codebase.

Server-side this is a new file `src/Ccgnf.Rest/Tournaments/
TournamentRunner.cs`; the benchmark harness becomes a shared
library under `src/Ccgnf.Bots/Bench/` that both CI and the REST
tournament endpoint call into.

### 10.2i — Benchmark harness

`tests/Ccgnf.Bots.Benchmarks/` (console app, not xunit). Given a
fixture directory of deck pairs + seeds, runs Fixed vs Utility and
Utility vs Utility for N games each, prints win-rate + avg turn
count, exits non-zero if any pair regresses from a checked-in
baseline JSON. Wire to `make bench-ai`. CI uploads the resulting
JSON as an artifact; the checked-in baseline is updated by hand when
a weight change is intentional (prevents silent regressions).

Initial fixture set: 8 pairs from §10f of
[10-long-term-ai.md](10-long-term-ai.md). Target thresholds
unchanged from that file — the concrete numbers are strict upper
bounds, not goals. A failing bench is a signal to investigate, not a
hard block on merge.

## Deterministic tie-breaking

Two actions can score identically. Rather than pick index 0 (bias
toward LegalActions ordering, which itself depends on engine-internal
iteration), break ties by the hash of
`(roomSeed, pending.Prompt, action.Kind, action.Label, State.StepCount)`.
Same seed + same pending + same state = same pick. This matches the
reference Utility's `DeterministicNoise` construction at
`reference-code/utilityAI/UtilityScorer.cs:366`, scaled to zero so it
functions as a tiebreaker rather than a scoring perturbation.

## Won't do in 10.2 (deferred to step 11+)

- **Multi-ply search.** Single-ply threat avoidance is the only
  non-reactive consideration allowed.
- **Learned weights.** Weights are hand-tuned JSON only.
- **Interrupts / replacement reactions.** The bot always passes on
  replacement-timing priority until those mechanics are in the engine.
- **Mulligan intelligence.** `MulliganConsideration` is a stub that
  keeps the current "always pass" behaviour; revisit once the
  Resonance Field + Banner interaction has settled.
- **Godot embedding.** `Ccgnf.Bots` targets netstandard2.1 to stay
  Godot-compatible, but the Godot host wiring lands with
  [step 9c](09-godot-client.md) rather than here.

## Web editor inside the preview console

The preview workflow (`mcp__Claude_Preview__preview_*`) is how every
step verifies behaviour; an AI weight/BT editor that doesn't live
inside the same Vite bundle is friction we don't need.

### Authoring model — not raw knobs

Raw weight tables + Hermite tangent slopes are unintuitive because
they force the author to reason in a mathematical vocabulary that
doesn't map to how a playtester actually thinks about AI behaviour.
The editor hides that layer behind four more-human abstractions, and
only exposes the raw JSON as an "advanced" escape hatch.

1. **Goal dials, not individual weights.** The top of `#/ai/weights`
   exposes 4 sliders that map to meaningful behavioural axes:

   | Dial          | Shifts these considerations                    |
   |---------------|------------------------------------------------|
   | Aggression    | `opponent_priority` ↑, `conduit_softness` ↑, `threat_avoidance` ↓ |
   | Tempo focus   | `on_curve` ↑, `tempo_per_aether` ↑, `clear_hand` ↑               |
   | Defence       | `threat_avoidance` ↑, `lowest_live_hp` ↑ (friendly units rise)   |
   | Resonance-chase | `resonance_alignment` ↑, `clear_hand` ↓                      |

   Each dial is a `[-1, +1]` float that the UI blends into the
   per-intent weight table through a fixed mapping matrix. Moving
   "Aggression" from 0 to +0.5 bumps three weights at once; authors
   stop thinking "what number should `threat_avoidance` be" and
   start thinking "I want this bot to be more aggressive."

2. **Curve presets, not knot handles.** The curve editor's primary
   UI is five named shapes — `Linear`, `Ease-in`, `Ease-out`,
   `Threshold`, `Bell` — selectable from a row of thumbnails. Each
   preset expands to pre-authored Hermite knots. An "Advanced"
   toggle reveals the knot drag handles for the 5% of cases
   presets can't express. This is the same pattern Unity + Godot
   use for their animation curves — expose named shapes by default,
   expose tangents only when asked.

3. **Natural-language labels on numeric inputs.** When raw weights
   are shown (because the author opened Advanced), label them with
   breakpoint words alongside the number: `0.0 = ignored`,
   `0.5 = minor`, `1.0 = normal`, `2.0 = emphasised`,
   `3.0 = dominant`. The label updates live as the author types.
   Replaces "what's a good number?" with "what importance?"

4. **Decision replay panel, always visible.** The editor permanently
   renders the last 10 `CpuDecision` frames from the most recent
   room as a table: `pending prompt → chosen action → top score`.
   Editing any dial / weight / curve re-runs the scorer against
   those 10 frames and highlights rows whose pick changed in red.
   Authors see "these 3 past decisions would now be different"
   directly, without playing a new game. This is what makes dials
   and presets honest — it closes the loop between a UI change
   and an observable behavioural change.

### Routing

New route `#/ai` with three sub-tabs:

- `#/ai/tune` — the goal-dial + curve-preset authoring surface
  (the four items above). Default landing tab. Saves to
  `encoding/ai/utility-weights.json`.
- `#/ai/advanced` — raw JSON textarea + schema-validated editor.
  Exposes the per-intent weight table and Hermite knots directly.
  Used for tuning beyond what dials can express, and for diffing
  two stable sets side by side.
- `#/ai/bt` — a read-only tree view of `encoding/ai/phase-bt.json`
  at first. Full authoring lands later as a follow-up; a JSON
  textarea with validation (schema mirrored from `BtNode`) is the
  v1 editor so the preview has round-trip ability without a
  drag-drop canvas.

### Backend surface (new REST endpoints)

- `GET /api/ai/weights` — read the current file, return as JSON.
- `PUT /api/ai/weights` — validated write. Rejects unknown
  consideration keys with `400` + a list of allowed keys. Triggers
  a hot reload of `UtilityBot`'s cached `WeightTables`.
- `GET /api/ai/phase-bt` / `PUT /api/ai/phase-bt` — same shape for
  the BT JSON; `PUT` validates by constructing a `BtNode` tree and
  rejecting on deserialisation failure.
- `POST /api/ai/preview-score` — body: `{ "decisionFrameId": "...",
  "weights": { ... } }`. Returns the top-3 re-scored under the
  supplied (not-yet-saved) weights, so the editor can show the
  delta before the author commits. No side effects.
- `GET /api/ai/bots` — list bot kinds available
  (`fixed`, `utility`, plus any experimental ones — see §"Production
   vs WIP sets" below). Used to populate the lobby's "CPU opponent"
  dropdown.
- `POST /api/ai/tournament` — run the round-robin described in
  §10.2h. Body: `{ "deckId": "...", "games": 20,
  "extraPairs": [{ "deckId": "...", "botProfile": "..." }, ...] }`.
  Streams `TournamentProgress` SSE frames; final frame is
  `TournamentResult` with the sorted leaderboard. Writes the result
  to `logs/tournament-<deckId>-<timestamp>.json`.

All `PUT`/`POST` endpoints are gated behind `CCGNF_AI_EDITOR=1`
by default. Production deployments don't expose weight editing or
tournament runs.

### Live preview integration

The existing Play tabletop already renders `RoomEventFrame`s in its
right column (§10.2f). The AI editor reuses that panel: when you have
both `#/ai/weights` and a `#/play/tabletop/{id}` open as split panes
(the preview supports multi-pane via `preview_resize` + separate
navigation), edits to weights trigger `POST /api/ai/preview-score`
for the most recent `CpuDecision` frame and render a before/after
comparison row inline in the tabletop's right column.

The one workflow this unlocks: a playtester sees the AI make a
dubious move, opens `#/ai/weights`, bumps `threat_avoidance` up
0.3, and watches the dubious move score drop below the alternative
— all without leaving the preview. If they like the change, **Save**
writes the file.

## Production-ready vs WIP / evolutionary AI sets

A single `utility-weights.json` collapses two very different
concerns: the weights we *ship* (regressed, hand-tuned, predictable)
and the weights we're *iterating on* (self-play experiments, learned
candidates, failed experiments we want to keep for post-mortem).
Storing both in one file forces an awkward choice between "break
production to try something" and "never try anything."

### Layout

```
encoding/ai/
  README.md
  stable/                           Production-ready. Referenced by default.
    v1-baseline.weights.json
    v2-defensive-tune.weights.json
    current.weights.json            Symlink / pointer to the active stable set
    phase-bt.json                   Shipped BT shell
    CHANGELOG.md
  experimental/                     WIP. Ignored by production builds.
    <timestamp>-<author>-<desc>/
      weights.json
      phase-bt.json                 (optional, overrides stable shell)
      bench.json                    Last bench result — self-play vs current stable
      notes.md                      Author's running log; becomes PR body if promoted
      lineage.jsonl                 Append-only parent-set history (TreeLog format)
    _selfplay/
      gen-<n>/                      Auto-generated by the self-play harness
        <hash>.weights.json
        bench.json
  schemas/
    weights.schema.json             JSON Schema for validation
    phase-bt.schema.json
```

`stable/current.weights.json` is what `UtilityBot` loads by default;
an experimental set is loaded only when explicitly named in the room
config or the `#/ai` editor. Production bundles strip
`experimental/` so the deployed host cannot accidentally run an
unfinished weight set.

### Room-level selection

Room creation accepts an optional `aiProfile`:

```json
{
  "seats": [{ "kind": "cpu", "aiProfile": "experimental/2026-04-19-self-play/gen-7/a3f2" }],
  "deck": "ember-aggro"
}
```

Unresolved profiles fall back to `stable/current`. A tabletop UI
badge shows the active profile so playtesters can tell stable from
experimental at a glance.

### Lineage and TreeLog

Directly mirrors the reference `TreeLog.cs` pattern. Every time a
weight set is saved — whether from the editor, a self-play
generation, or a hand-commit — a record is appended to the
experiment's `lineage.jsonl`:

```json
{ "name": "gen-7-a3f2", "generation": 7, "timestamp": "2026-04-19T18:22:03Z",
  "parentName": "gen-6-b91c", "tags": { "source": "selfplay", "winrate": "0.58" },
  "weights": { ... } }
```

The file format is identical to `TreeLog`'s record shape so
downstream tooling (a future visualiser, CSV export, spreadsheet
scratch pad) can consume both the BT history and the weight history
with one reader.

### Promotion workflow

`make ai-promote SRC=experimental/2026-04-19-.../gen-7/a3f2` runs:

1. Validate the weight file against `schemas/weights.schema.json`.
2. Run `make bench-ai` with the candidate as the CPU profile. Fail
   if the candidate loses ≥ 45% mirror matches against
   `stable/current`.
3. Copy the file to `stable/v{N+1}-<slug>.weights.json`, bump the
   `current` pointer, append to `CHANGELOG.md` with the bench JSON
   summary.
4. Stage the changes for a human commit. The author writes the
   message; the script never commits on its own.

This is the only path from experimental to stable. No editor
shortcut to `stable/` — the gate is the bench run, and the bench
run has to be reproducible from CI.

### Self-play harness (deferred sketch)

Not built in 10.2; called out so the directory layout supports it
when it lands. Idea: start from `stable/current`, perturb weights
by Gaussian noise scaled per consideration, play N mirror matches,
promote survivors by win-rate, repeat. Each generation writes under
`experimental/_selfplay/gen-<n>/` and appends to a top-level
`_selfplay/lineage.jsonl`. Everything still gates behind
`make ai-promote`.

## Decks × AI — affinity vs adaptation

Two defensible positions, each with real tradeoffs.

### Option A — one AI, adapts to any deck

The bot reads the deck's composition (faction mix, curve, keyword
density) when the room starts and mutates its own weight table at
runtime. `ResonanceAlignmentConsideration` already does a minor
version of this — it checks the card against the player's Banner.
Scaling up means adding deck-profile-derived weight modifiers:

- High-curve deck → `on_curve` weight halves (the deck can't play
  on-curve, so stop penalising off-curve plays).
- Mono-faction deck → `resonance_alignment` weight doubles (Banner
  triggers more reliably, lean into it).
- Heavy removal → `threat_avoidance` weight drops (we can clear
  threats post-hoc instead of pre-empting).

**Strengths.** One bot to tune. Playtests generalise. No per-deck
maintenance overhead.

**Weaknesses.** The weight surface blows up: the mutation function
itself becomes another layer to tune. A bug in the adaptation logic
is silent — the bot plays badly, the author blames the weights, the
real culprit is three layers up.

### Option B — AI × deck affinity; mismatch plays poorly

Each stable weight set carries a declared affinity:

```json
{
  "affinity": {
    "archetypes": ["aggro", "mono-ember"],
    "pairs_well_with": ["ember-aggro", "bulwark-rush"],
    "mismatch_penalty": 0.4
  }
}
```

When an AI profile is paired with a deck outside its affinity list,
`UtilityBot` applies a flat score damping (`0.4`) to every
consideration that relies on matched characteristics. The bot still
plays legally — it just plays worse. A defensive AI handed an aggro
deck loses on purpose, and that's fine because nobody should be
pairing them.

**Strengths.** The bot is honest. You can see at a glance from the
affinity block which decks a profile was tuned for. It makes the
per-deck tuning loop explicit and avoids the "but what does the AI
do with deck X" question — the answer is "poorly, here's why."

**Weaknesses.** Scales linearly with decks. Every new deck wants a
matched AI profile; playtesting budget is finite.

### Recommendation

**Ship Option B. Keep Option A available as a single "universal"
profile.**

Concretely:

1. Each stable weight set declares affinity. `ember-aggro` deck
   has a matched `v1-ember-aggro.weights.json`. Mismatched
   pairings take the penalty and are flagged in the lobby with a
   warning chip.
2. A single `v1-universal.weights.json` exists that lies somewhere
   on the Pareto frontier — it plays every deck at a mediocre
   level. This is the default for undeclared deck pairings.
3. The editor (`#/ai/weights`) shows the active affinity list and
   lets authors edit it alongside the weights. Editor-saved sets
   default to `affinity.archetypes = []` which forces explicit
   declaration before promotion.
4. Self-play experiments start from `v1-universal` when the deck
   is unknown, and from the matched profile when one exists. The
   harness records which starting profile it used in
   `lineage.jsonl`.
5. The adaptation strategy from Option A is reserved for a later
   iteration — prototyped as a single consideration
   (`DeckAdaptiveConsideration`) layered on top of a base set
   rather than a separate system. If it outperforms per-deck
   profiles in benchmarks, it replaces them. Until then, the
   affinity approach is cheaper to build and easier to reason
   about.

The key property we're buying: every AI-deck pair has a predictable
quality level, and that level is either "matched" (hand-tuned),
"universal" (mediocre on purpose), or "mismatched" (intentionally
poor). No surprise regressions, no "why did the bot play aggro
when it has a control deck" incidents.

## Resolved in this revision

- **Weight/curve authoring ergonomics.** Raw knobs out; goal dials,
  curve presets, natural-language labels, and a persistent decision-
  replay panel in. Advanced JSON remains available under `#/ai/
  advanced` for cases presets don't cover. See §"Authoring model —
  not raw knobs" above.
- **Sticky intent.** Promoted from "don't pre-build it" to phase
  10.2g. Lives on the `Room`, biases the BT Selector by 1.5× for
  up to N=5 decisions with 0.1-per-decision decay, resets on phase
  change, short-circuits on `lethal_check` and defensive-intent
  triggers. Tunable via `phase-bt.json → stickiness`.
- **Deck × AI pairing.** Option B confirmed. Deck creation gets an
  `archetypes: [...]` tag (chip input, curated vocabulary). New
  `#/decks` "Suggest an AI" button runs a mirror round-robin
  tournament (`POST /api/ai/tournament`) with optional
  `extraPairs` for cross-deck baselines. Top result becomes the
  deck's `suggested_ai` + the bot's `pairs_well_with` entry. See
  phase 10.2h.
- **Self-play determinism.** Record the perturbation RNG seed in
  each `lineage.jsonl` record as a top-level `rngSeed` field
  alongside the existing `name`/`generation`/`parentName`. Without
  it, a generation can't be reproduced; with it, any past
  experimental set can be regenerated from its parent + seed.

## Open questions still unresolved

- **Affinity inference.** Could the deck-profile → archetype mapping
  be derived automatically from the card list instead of hand-
  authored? Probably yes for faction + curve; probably no for
  strategic archetype (control vs midrange vs tempo). Defer until a
  playtester asks. The tournament in 10.2h arguably makes this
  question moot — instead of inferring the tag, let the author hit
  the button and let win-rate declare the matchup.
- **Tournament cost.** 20-game mirror × 4 bots = 80 playthroughs. At
  current engine speeds that's fine; if bot count or game length
  balloons, the tournament endpoint needs progress checkpoints so a
  user refresh doesn't lose mid-run results. Revisit when a
  tournament takes > 30s.
- **Curve preset library.** Starting with five named shapes
  (Linear / Ease-in / Ease-out / Threshold / Bell). If a sixth
  recurring shape shows up in advanced-mode hand edits, canonicalise
  it. Resist adding presets preemptively.
